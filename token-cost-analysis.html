Token-cost analysis

Does the annotate skill cost more tokens than vanilla Claude Code?

Short answer: it depends on whether you iterate. One-shot = more. Back-and-forth = less.

There is no single number. Annotate's cost flips on how many times you interact with a response.

  • One-shot response (read it, move on) — ~30–60% more  You pay setup + JSON wrapping with nothing to amortize them against.
  • Iterative session (comment → revise → repeat) — cheaper, and it compounds  Break-even ≈ 4–8 comments; ~50% cheaper by 15+.
  • One fixed tax — the first time annotate triggers per session it loads its SKILL.md (~7.4k tokens) into context. Paid once, then cached.

The re-read is the dominant input cost: the server has no block-update endpoint, so every comment pulls the whole document back into context. Prompt caching makes this cheap only if you click within ~5 min (cache TTL).

The two regimes

One-shot

Vanilla wins

Vanilla just emits text and stops. Annotate adds fixed setup + wrapping with no interactions to spread it over. ~30–60% more.

Iterative

Annotate wins

A click is an unambiguous reference. Terminal iteration re-emits context, re-prints state, and pays a “which paragraph?” tax. Annotate's per-edit cost stays flat.

The win isn't the browser UI — it's per-edit compactness. Editing one block sends one block's diff; a terminal turn that “responds clearly” repeats surrounding context, and users ask for full re-prints to see current state. Annotate moves that state into the browser, so the re-emission tax disappears. Diagrams amplify the win: the SVG renders client-side and never enters context.

Caveats to state honestly

  • Cache-dependent. Slow reader (>5 min between clicks) → the blocks.json re-read is billed at full price, roughly tripling per-comment input and pushing break-even out to ~12 comments. Annotate still wins on long sessions.
  • Estimates ±20%. Actual cost varies with model verbosity and conversation history.

Token figures are model-independent; billable comparisons assume Opus-class pricing where output ≈ 5× input (billable ≈ output×5 + input). Cache reads bill at ~10% of input. Figures derived from skills/annotate/SKILL.md and server.py.
