Coordination for parallel Claude Code

One master. Many workers. One queue.

Drive a fleet of Claude Code sessions from a single seat. The master is a task manager; each worker owns one task in its own worktree. They coordinate through a small SQLite queue — no tty tricks, no shared memory. Idle workers cost zero tokens.

The mental model

Master — where you sit

The session you drive from. It never does task work itself — it watches the board, queues commands, and collects results. Keep it for steering.

Worker — parked on a task

A session living in a task's worktree, listening. It executes whatever the master queues, at full autonomy, then reports back — and stays quiet until then.

Command matrix

Only /mesh-dispatch takes arguments
Setuponce per machine
/mesh-init no args First run on a machine, or when a skill reports “database missing.” Creates the SQLite store. Idempotent.
Workerrun inside the task's session
/mesh-join no args The one command you type in a worker. Registers this session and arms the doorbell in one step. After it, the session listens.
/mesh-arm no args Re-arm the doorbell without re-registering. Rarely needed directly — /mesh-join already arms.
/mesh-poll legacy One poll tick for the old /loop 1m /mesh-poll model. Superseded by the event-driven doorbell; kept for back-compat.
Masterrun in the driving session
/mesh-board no args “What's running?” Live roster: each worker's task, alive / stale / dead, current activity, and recent commands.
/mesh-dispatch args Queue work for a worker — or all of them. The only command that takes arguments (see below).
/mesh-await no args After dispatching, wait quietly to be woken the moment a result lands. No polling, zero tokens while waiting.
/mesh-collect no args One-shot: pull any already-finished results right now, no waiting. An instant inbox check.
/mesh-pause no args Emergency-stop the whole fleet. Workers keep heartbeating (stay green) but execute nothing.
/mesh-resume no args Lift the pause; workers resume executing queued commands.

Anatomy of a dispatch

/mesh-dispatch PMP-211 shell "npm test"

target

Who runs it: a task label (PMP-211, reporting-openapi), a session id, or * to broadcast to everyone.

kind

shell runs a command in the worktree · prompt hands the worker a task · control takes it offline (stop).

payload

The quoted command, prompt, or verb. Free text — it may even contain |.

Typical flows

1

Stand up a worker

# in a session opened in the task's worktree
/mesh-join   # registers + listens. leave it be — free while idle.
2

Drive a worker from the master

/mesh-board                             # see who's live
/mesh-dispatch PMP-211 shell "npm test"   # queue work
/mesh-await                             # get woken when it finishes
3

Broadcast to everyone

/mesh-dispatch * prompt "pull main, rebase your branch, report conflicts"
4

Panic button

/mesh-pause    # everything holds
/mesh-resume   # ...back to work
5

Retire a worker

/mesh-dispatch reporting-openapi control stop

Good to know

Full autonomy

Workers execute what they claim, unattended. The global pause is the only stop. Dispatch deliberately.

Don't drive a hand-typed session

Arm workers in dedicated sessions; keep the one you're typing in as the master.

The doorbell

Detection runs in a detached shell that blocks until real work, then exits — waking the model exactly once. Idle cost: zero. /mesh-await is its master-side twin.

Crash recovery

A claimed command whose worker dies is auto-requeued (the reaper runs inside /mesh-board and /mesh-dispatch). No daemon, no manual step.
