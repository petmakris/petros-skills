Coordination for parallel Claude Code

One master. Many workers. One queue.

Drive a fleet of Claude Code sessions from a single seat. The mesh is two orthogonal layers: a transport that reaches and drives live sessions, and a task backlog that lives in the store, independent of any session. They coordinate through a small SQLite queue — no tty tricks, no shared memory. Idle workers cost zero tokens.

Two orthogonal layers

the two meet only at mesh_dispatch

Layer 1 — transport

Reach and drive live sessions: a roster (sessions), a command queue (commands), the doorbell, and the reaper. Task-agnostic — it knows nothing about tasks.

Layer 2 — task manager

A backlog (tasks + task_sessions, schema v3) that lives in the store, independent of sessions. A task is serviced by 0..N sessions — 0: the master does it inline or via a subagent · 1: one worker · 2–3: several on the same task.

The mental model

Master — where you sit

The session you drive from, and that owns the backlog. It watches the board, queues commands, and collects results — and can service a task itself, inline or via a subagent.

Worker — a live session

A session addressed by a free-form label, listening in a worktree. It executes whatever the master queues, at full autonomy, then reports back — and stays quiet until then.

Command matrix

Only /mesh-dispatch takes arguments
Setuponce per machine
/mesh-init no args First run on a machine, or when a skill reports “database missing.” Creates the SQLite store. Idempotent.
Workerrun inside the task's session
/mesh-join <label> label The one command you type in a worker. Registers this session under a free-form label (auto-derived from the branch if omitted) and arms the doorbell in one step. After it, the session listens.
/mesh-arm no args Re-arm the doorbell without re-registering. Rarely needed directly — /mesh-join already arms.
/mesh-poll legacy One poll tick for the old /loop 1m /mesh-poll model. Superseded by the event-driven doorbell; kept for back-compat.
Masterrun in the driving session
/mesh-board no args “What's running?” Live roster — each worker, alive / stale / dead, current activity, recent commands — plus the task backlog grouped by status.
/mesh-dispatch args Queue work for a worker — or all of them. The only command that takes arguments (see below).
/mesh-await no args After dispatching, wait quietly to be woken the moment a result lands. No polling, zero tokens while waiting.
/mesh-collect no args One-shot: pull any already-finished results right now, no waiting. An instant inbox check.
/mesh-pause no args Emergency-stop the whole fleet. Workers keep heartbeating (stay green) but execute nothing.
/mesh-resume no args Lift the pause; workers resume executing queued commands.

Anatomy of a dispatch

/mesh-dispatch docs-rewrite shell "npm test"

target

Who runs it: a free-form label (docs-rewrite, db-migration, or PMP-211 if you like — no Jira coupling), a session id, or * to broadcast to everyone.

kind

shell runs a command in the worktree · prompt hands the worker a task · control takes it offline (stop).

payload

The quoted command, prompt, or verb. Free text — it may even contain |.

Step by step

what you type, and what you see — one lane per terminal
TYPE a command you type in the terminal ASK something you tell Claude in plain English SEE what appears back
1 · First run — start the mesh and connect a worker — Do this once. The left terminal becomes your master; a second terminal joins as a worker it can drive. /mesh-init is only needed the very first time on a machine.
2 · Give a worker a job and wait for the result — From the master you queue a command, then /mesh-await wakes you the moment it finishes — no polling, zero tokens while waiting.
3 · Work with tasks — add, check status, mark done — You don't type these — just tell Claude in plain English and it runs the task tools for you. Spawning a task's worker is automatic.

A guided first run

every call & response below is real — captured from an actual session

You drive with plain language

The task_* / mesh_* tools are MCP tools — you don't type them, you ask Claude and it makes the call. Each step shows the request in words, the tool call it becomes, and the exact JSON that came back. Paths are trimmed to …/mesh-worker-1 for width.

1

Put a task on the backlog

“Add a task spawn-1: create a file PROOF.txt containing mesh-spawn-ok, then confirm it exists.”

the call
task_add(slug="spawn-1", title="Auto-spawn live test",
         description="Create PROOF.txt containing mesh-spawn-ok, verify it exists.")
returns
{ "slug": "spawn-1", "status": "todo" }

A task exists before any session does. The backlog lives in the store, independent of workers.

2

Spawn a worker for it

“Spawn a background worker for spawn-1 in that directory.”

the call
task_spawn(slug="spawn-1", cwd="…/mesh-worker-1")
returns
{ "task": "spawn-1", "label": "spawn-1",
  "session_id": "5EBEFA89-7173-…",
  "status": "in_progress", "command_id": 1 }

One call did four things: launched a real claude --bg session, joined it to the mesh, assigned it the task, and dispatched the task as command #1. No hand-starting.

3

Watch the board

“What's running?”

the call
mesh_board()
returns (trimmed)
"sessions": [{ "label": "spawn-1", "liveness": "alive",
               "status": "idle", "cwd": "…/mesh-worker-1" }],
"commands": [{ "id": 1, "kind": "prompt", "state": "pending" }],
"tasks":    [{ "slug": "spawn-1", "status": "in_progress",
               "sessions": ["spawn-1"] }]

One JSON view of all three planes: who's live, what's queued, and the backlog. The worker is booting, so command #1 is still pending.

4

Collect the result

The worker claims #1, does the work, and reports back. “Any results in?”

the call
mesh_collect()
returns
{ "id": 1, "label": "spawn-1", "exit_state": "ok",
  "output": "Created PROOF.txt (mesh-spawn-ok) at
             …/mesh-worker-1/PROOF.txt; verified it exists." }

The worker's summary comes back as the command's output. Completing the dispatched task auto-advances spawn-1 → done. Prefer /mesh-await to be woken the instant it lands — zero tokens while waiting.

5

Ask a live worker anything

“Ask spawn-1 for its status and whether PROOF.txt still exists.”

the call
mesh_ask(target="spawn-1", question="status? does PROOF.txt still exist?")
→ { "command_ids": [2] }   then  mesh_collect()
returns
{ "id": 2, "label": "spawn-1", "exit_state": "ok",
  "output": "Idle and armed after completing spawn-1;
             PROOF.txt still exists with contents 'mesh-spawn-ok'." }

An ask is just a prompt whose answer returns as output — status, a decision, next move — without handing over a task.

6

Pause & resume the fleet

“Hold everything.” A dispatched command now waits instead of running.

the calls
mesh_pause()   → { "result": "paused" }
# dispatch a command now — it sits at "pending" and does not run
mesh_resume()  → { "result": "resumed" }   # the held command is claimed & runs

The global pause is the only stop. Workers keep heartbeating (stay green) but claim nothing until you resume.

7

Crash recovery — automatic

A worker dies while holding a running command. Nothing is lost.

what happens
# the reaper runs automatically inside mesh_board and mesh_dispatch
# a running command whose worker's pid is gone is returned to the queue:
   command 4:  running  →  pending   # re-claimable by another worker

No daemon, no manual step. A killed worker's in-flight work is requeued the next time you look at the board or dispatch.

Hand-driven equivalents

Prefer to type? The same moves exist as slash skills for hand-started workers: /mesh-join in a worker, then /mesh-board · /mesh-dispatch <label> shell "npm test" · /mesh-await · /mesh-dispatch * prompt "…" (broadcast) · /mesh-dispatch <label> control stop (retire) from the master.

Good to know

Full autonomy

Workers execute what they claim, unattended. The global pause is the only stop. Dispatch deliberately.

Don't drive a hand-typed session

Arm workers in dedicated sessions; keep the one you're typing in as the master.

The doorbell

Detection runs in a detached shell that blocks until real work, then exits — waking the model exactly once. Idle cost: zero. /mesh-await is its master-side twin.

Crash recovery

A claimed command whose worker dies is auto-requeued (the reaper runs inside /mesh-board and /mesh-dispatch). No daemon, no manual step.

The MCP tools

shipped — you invoke these by asking Claude, not as /slash commands

The task-manager redesign is built (Phases 1–3), delivered as typed MCP tools rather than more slash-commands. The plugin declares a stdio MCP server (session-mesh, Python/FastMCP, launched with uv run --script — no install); every session spawns its own instance over the shared SQLite DB. Each tool is a thin adapter that shells into mesh.sh, so the shell stays the single source of truth. The DB is the hub — no daemon.

Layer 1 · transporttyped twins of the /mesh-* skills
mesh_board MCP Sessions + liveness, recent commands, and the task backlog — one JSON view.
mesh_dispatch MCP Queue work for a target (label · session id · *). The one seam where Layer 2 reaches Layer 1.
mesh_collect MCP Pull newly-finished command results and mark them collected.
mesh_ask MCP Ask a worker for status / next move — the answer returns as the command's output.
mesh_pause MCP Emergency-stop the whole fleet.
mesh_resume MCP Lift the pause; workers resume executing queued commands.
Layer 2 · task managera backlog independent of sessions — schema v3
task_add MCP Add a task to the backlog — before any session exists.
task_list MCP The backlog, optionally filtered by status; each task shows its assigned sessions.
task_get MCP One task, with its assignments.
task_assign MCP Link a session to a task. One active task per session — pass force to move it.
task_unassign MCP Unlink a session from a task.
task_set_status MCP Advance a task's lifecycle: todo · in_progress · blocked · done.
task_done MCP Shorthand to mark a task done.
task_spawn MCP Auto-spawn a claude --bg worker for a task — it joins the mesh, is assigned, and gets the task dispatched. No hand-starting.
task_ask MCP Ask every worker on a task for status / next move.

MCP is for acting; the doorbell is for receiving

The doorbell (/mesh-arm, /mesh-await) is deliberately not an MCP tool — a blocking MCP call would hold the model's turn open. The background watcher stays token-free.

The live web board

a third front door — for when you'd rather see it than read it

Prefer a dashboard to the text board? Run /mesh-ui and a local page opens with the whole mesh at a glance: the worker roster (alive / stale dots, current activity, cwd·pid), the task backlog as a Todo / In progress / Blocked / Done kanban, and the recent-command feed (click a finished command to read its output). Buttons add tasks, spawn workers, pause/resume the fleet; click any task for a detail drawer that reassigns workers and moves it through its lifecycle.

How it's builtsame single source of truth
session-mesh/ui/server.py stdlib A dependency-free HTTP server (uv run --script). Holds no state: every read/write shells into mesh.sh — the same functions the MCP tools call. Serves the page and a small JSON API.
GET /api/board read The board snapshot (mesh_board_json + the paused flag). The page also opens /api/stream — Server-Sent Events that push a fresh board the instant anything changes, falling back to a 1.5s poll if the stream drops.
POST /api/action write Every button routes here, to a hard-whitelisted map of ops → mesh.sh functions (pause/resume, task add/status/done/assign/spawn, dispatch, ask). The UI is a front door, not a second implementation.

Localhost only

It binds to 127.0.0.1 and drives autonomous workers — so it must never be reachable off the machine. No auth, because nothing else can reach it.

Launch & stop

/mesh-ui opens it (default port 8787, MESH_UI_PORT overrides). Stop with pkill -f session-mesh/ui/server.py. It self-inits the DB, like the MCP server.
